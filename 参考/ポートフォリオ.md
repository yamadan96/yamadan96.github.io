# 2. AIエンジニアになりたいと思ったきっかけ、課題及び対策

## 2.1 エンジニアを志したきっかけ

高校3年生の時、孫正義氏のシンギュラリティに関する講演を聞いたことが、私の人生に大きな影響を与えました。AIが社会を根本から変革し、未来の可能性を大きく広げるという話に強く魅了され、その時からAI技術に深い関心を持つようになりました。

私はもともと数学や物理が好きで、論理的思考や数理的アプローチに対して強い興味と得意意識があります。AIの分野はまさに、これらの素養を活かしながら、複雑な課題を数式やアルゴリズムで解決していくフィールドであり、自分に最適な職業だと確信しています。

また、AIは特定の業界に限定されず、医療、製造、金融、教育など多様な分野で応用できる汎用性の高さも魅力的でした。私は単に技術を扱うだけでなく、様々な業界や社会課題に対してAIを活用し、実際に人々の生活をより良く変えていきたいと考えています。

これまで、スタートアップ企業でのAI関連インターンを通じて、実際にLLMや画像処理などのAIプロジェクトに携わり、理論だけでなく実務レベルでのAI技術の応用を経験してきました。こうした経験を踏まえ、私はAIエンジニアとしてテクノロジーの力で社会に大きなインパクトを与えたいという強い意志を持っています。

## **2.2 ソフトウェアエンジニアとしての課題**

私の課題は、コンピュータサイエンスの基礎を体系的に学んでこなかったことです。
私は電気電子情報工学科出身であり、ハードウェア設計や信号処理などの物理寄りの領域を中心に学んできました。そのため、情報学科で扱うようなデータベース、ネットワーク、OS、データ構造・アルゴリズム、システムデザインといった基礎的な領域を包括的に習得する機会が限られていました。
AIや機械学習の実装力には自信がある一方で、よりスケーラブルで堅牢なシステムを設計・運用するための基盤的な理解が不足していると感じています。

## **2.3 課題に対する行動・取り組み**

この課題に対して、理論と実務の両面から基礎を強化する取り組みを行っています。
理論面では、オンライン学習サービス「Recursion」を活用し、コンピュータサイエンスの基礎から応用まで体系的に学んでいます。特に、データ構造とアルゴリズム、ネットワーク、OS、データベース設計など、これまで大学では触れられなかった領域を重点的に学習し、理解をコードレベルで定着させています。
また、実務ではAIスタートアップでの開発を通じて、学んだ理論をAPI設計やMLOps構築に応用し、よりスケーラブルで堅牢なシステム設計を意識して実装しています。
このように、理論のインプットと実務でのアウトプットを並行して進めることで、AI技術を支えるCS基盤の理解を着実に深めています。

## **3.1.  被災家屋の損傷度画像分類AIの開発（**ファインチューニング手法に基づく被災建物画像の多クラス損傷度分類の精度改善**）**

https://www.ipsj.or.jp/event/fit/fit2025/index.html

山田悠人, 鈴木海友, 松田一朗, 多賀祥平, 松澤智史, 柏田 仁, 二瓶泰雄:

「ファインチューニング手法に基づく被災建物画像の多クラス損傷度分類の精度改善」, 第24回情報科学技術フォーラム(FIT 2025)講演論文集, Vol.3, No.I-029, pp.339-340, (2025年9月).

https://www.ieice.org/publications/conference-FIT-DVDs/FIT2025/data/html/program/pdf/I-029.pdf

[I-029.pdf](attachment:625ee5cf-26af-43fa-80df-6de4d18c9f46:I-029.pdf)

地震・津波などの大規模災害直後における「被災建物の損傷度分類」の自動化に取り組みました。現状は専門家の目視に依存しており、迅速性・客観性・人的コストの面で課題があります。そこで本研究では、ResNet・ConvNeXt・ViTなど異なるアーキテクチャの事前学習済みモデルを用いて比較検証し、それぞれの特徴と精度傾向を明らかにしました。さらに、パラメータ効率に優れるLoRA（Low-Rank Adaptation）を導入することで、少数データ環境でも高精度なファインチューニングを可能にし、分類精度を段階的に向上させました。従来の全パラメータ更新型手法を上回る結果を得ており、モデル軽量性と汎化性能の両立も達成しています。これらの成果は、災害時の迅速な状況把握や復旧支援に資する技術として高い実用性を持ち、FIT情報科学技術フォーラム2025での発表済み。

## **3.2. 長期文脈を活用したフレーム外挿モデルによる動画異常検知**

事故などの異常を動画から検知するタスクにおいて、従来手法が苦手とする長期的な背景変化や動作パターンの抽出を目的に、Clockwork Variational Autoencoder（CW-VAE）を活用した手法を提案しました。OOPS!データセットを用いて、ConvLSTMとの定量比較を実施し、AUC・F1など複数指標で優れた結果を得ました。CW-VAEの時間的抽象化機構により、短期・長期双方の変化を捉えることができ、動画異常検知における時間スケール分離の有効性を示すことができました。

## **3.3. マルチモーダルRAGによる日本語専門文書の要約・分析システム開発（【第二・三期】東京大学 松尾・岩澤研究室 LLM Community "LLMATCHプログラム"）**

現在、東京大学 松尾研究室主催の選抜制プログラム「LLMATCH」において、図表や画像を含む日本語専門文書を対象とした、マルチモーダルな情報検索・要約が可能なRAGシステムの開発に取り組んでいます。CLIPやLangChainを活用し、画像・表・テキストの埋め込み統合や生成系LLMとの連携による検索拡張生成の仕組みを設計中です。私はシステムの設計・実装・評価指標の検討を担当しており、現在はPoC構築に向けてチームでの検証と技術的議論を重ねています。社会実装を見据えた設計力とマルチモーダルAIへの理解を深めている最中です。

## 3.4 マルチタスク学習を用いた被災家屋6クラス損傷度分類AI

概要：
少量・不均衡データの被災家屋画像に対し、LoRA適用モデルとマルチタスク学習を組み合わせて6クラス分類性能を改善した。
研究意義：
専門家依存の被災判定を自動化し、災害時の迅速・正確な状況把握と復旧判断に寄与する技術基盤を構築できる。
工夫点：
DINOv2・ConvNeXt 等の比較とLoRA微調整に加え、コスト行列導入と補助タスク（タイプ/重症度）を用いたMTLで汎化性能と安定性を向上させた。
自身の貢献：
前処理、LoRA設計、MTL損失設計、学習パイプライン構築、比較評価、考察まで全工程を主導した。

## 3.5 **連続変換生成モデルNFDMのTransformerベース簡略化に関する基礎検討**

Neural Flow Diffusion Models（NFDM）を基盤とし、Just Image Transformerベースへ拡張・簡略化する研究に取り組んでいます。NFDMは連続変換を扱える汎用性の高い生成モデルで、改良次第で多様なタスクへの応用が期待されます。本研究では、モデル構造の単純化とTransformer化による挙動整理を目的とした予備実験を行い、基礎知見の獲得を目指しています。

### 4.7.1 **画像キャプション生成の高速化（Vision-Language Model）**

**役割**：実装・実験設計・評価

**目的**

同じ画像から複数のキャプション（説明文）を生成する際、Vision Encoder部分をキャッシュしておき再利用することで、処理時間を大幅に短縮する。

**背景・課題**

- 家族アルバムの画像に対して、複数のキャプションを自動生成するサービスを開発中
- 従来は、キャプションを生成するたびに画像を1から処理していたため、時間がかかっていた
- 画像処理（Vision Encoder）が全体の処理時間の約35%を占めており、ボトルネックになっていた
- ユーザー数の増加に対応するため、処理速度の向上が必要

**成果（定量）**

- 全体の処理時間を**42.8%短縮**（2Bモデル）、**28.1%短縮**（4Bモデル）
- 画像処理部分：**3.2秒 → 0.001秒（99.97%削減）**
- キャプションの品質：**劣化なし**（同じ設定で比較検証）
- GPU使用時と比較して、CPUでも**15.4倍高速**に処理可能（ベクトル再利用時）

**主な取り組み**

1. **画像処理結果の抽出・保存**
    - 画像処理モデル（Vision Encoder）の出力を直接取得
    - 処理済みの画像データ（ベクトル）を保存して再利用可能に
2. **ベクトル再利用の仕組み構築**
    - 保存したベクトルを使って、2回目以降の処理をスキップ
    - キャプション生成部分のみを実行することで、大幅な時間短縮を実現
3. **処理時間の詳細分析**
    - どの処理にどれくらい時間がかかっているかを細かく計測
    - ボトルネックを特定し、最適化の方針を明確化
4. **デモアプリによる検証**
    - 社内向けのデモアプリを作成し、ベクトル再利用の効果を視覚的に確認
    - 速度と品質の両面で問題ないことを検証
5. **段階的な検証**
    - 1枚の画像 → 複数の画像 → 本番環境の順で、段階的にテストを実施

**技術スタック**

Qwen3-VL（Vision-Language Model）, PyTorch, Transformers, Google Cloud Vertex AI（V100 GPU）, Python 3.10, Gradio

**工夫したポイント**

- 段階的なテストで、安全性を確保しながら実装

**事業への貢献**

- 同じ画像に対して複数のキャプションを生成する際の**運用コストを大幅削減**

- 将来的なユーザー数の増加にも対応できる基盤を構築

### 4.7.2 **機械学習パイプラインのコード整理（重複処理の削除）**

**役割**：設計・実装・テスト 主担当

**目的**

複数の画像解析プログラムに散らばっていた同じ処理を1箇所にまとめ、コードの重複を削除して保守性を向上させる。

**背景・課題**

- 顔抽出、感情推定、頭部姿勢推定など、複数のプログラムで同じフィルタリング処理が重複していた
- 「同じコードを複数箇所に書かない」という原則（DRY原則）に違反
- 処理ロジックを変更する際、複数のファイルを修正する必要があり、ミスが発生しやすかった
- エラーチェックの処理が複雑で、コードが読みにくかった

**成果（定量）**

- 重複していた処理を削除し、**コード量を約130行削減**（約50%削減）
- ルールベースの自動チェック機能を導入し、エラー検出を自動化
- テストコードも整理し、**約40%削減**（重複テストの統合）
- 本番環境で18時間監視し、**エラー・挙動の変化なし**を確認

**主な取り組み**

1. **処理の集約**
    - 上流のプログラム（face-extractor）に顔フィルタリング処理を集約
    - 下流のプログラム（emotion-estimator, head-pose-estimator）から重複コードを削除
2. **ルールベースの自動チェック**
    - 手作業のエラーチェック（if文の連続）を、ルールベースの自動チェックに置き換え
    - 「顔のサイズは20px以上」「検出スコアは0.5以上」などのルールを定義
    - ルール違反があれば自動的にエラーを検出
3. **テストの自動化**
    - パラメータ化テストを導入し、様々なパターンを自動的にテスト
    - 境界値（最小値・最大値）や異常値のテストを体系的に実施
4. **段階的なリリース**
    - テスト環境で十分に検証してから、本番環境にデプロイ
    - 本番環境でも監視を継続し、問題がないことを確認

**技術スタック**

Python, Pydantic（データ検証ライブラリ）, pytest（テストフレームワーク）, Git, CI/CD, 監視ツール（Grafana, New Relic）

**工夫したポイント**

- 各プログラムの責任範囲を明確に分離（face-extractorは顔検出とフィルタリング、他は各種推定のみ）
- 手作業のチェックではなく、ルールを定義することで、コードを読みやすく保守しやすくした
- テストの自動化により、将来の変更でもバグを早期発見できる仕組みを構築
- コードの重複をなくすことで、変更時の修正箇所を最小化

**事業への貢献**

- コードの保守性向上により、新機能追加や変更の工数を削減
- バグの混入リスクを低減し、サービスの安定性を向上
- 他の機械学習サービスにも応用できる設計パターンを確立

**キーワード**

MLOps／コードリファクタリング／DRY原則（重複排除）／自動テスト／保守性向上

### 4.7.3 **メッセージキュー（SQS）のタイムアウト設定最適化**

**役割**：分析・設計・検証 主担当

**目的**

画像解析システムでエラーが発生した際の復旧時間を短縮するため、メッセージキューのタイムアウト設定を最適化する。

**背景・課題**

- 画像解析システムは、AWS SQS（メッセージキュー）を使って処理を分散
- エラー発生時、メッセージが再処理されるまでの待ち時間（Visibility Timeout）が900秒〜1200秒と長すぎた
- 実際の処理時間は平均0.35秒と非常に短く、設定値との乖離が大きかった
- エラー発生時、ユーザーが結果を受け取るまで**15分以上**かかっていた
- なぜこの設定値なのか、根拠が不明確だった

**取り組み**

1. **実際の処理時間を徹底調査**
    - New Relic（監視ツール）で30日間のデータを収集
    - 平均値、90パーセンタイル、99パーセンタイル、最大値などを算出
    - エラー率やスループット（処理量）も分析
2. **最適な設定値の算出**
    - 「最大処理時間 + 安全マージン」という設計方針を策定
    - 短すぎると重複処理が発生、長すぎると復旧が遅れるバランスを考慮
    - データに基づいて、900秒 → 65秒への短縮を提案
3. **テスト環境での検証**
    - 意図的にエラーを発生させて、再処理の挙動を確認
    - メッセージが重複して処理されても問題ないこと（冪等性）を検証
    - タイムアウトをランダムに発生させるテストツールを開発
4. **段階的な短縮計画の策定**
    - 一気に短縮するのではなく、65秒 → 40秒 → 25秒 → 20秒と段階的に短縮
    - 各段階で監視を行い、問題がないことを確認してから次のステップへ
5. **インフラ設定の準備**
    - Terraform（Infrastructure as Code）でキュー設定を管理
    - 設定変更後の監視項目（最古メッセージの待ち時間、エラーキューへの流入、重複処理数）を定義

**成果（定量）**

- 処理時間の実測：平均**0.35秒**, 99パーセンタイル**1秒以下**, 最大**60秒**, エラー率**0.01%未満**
- タイムアウト設定：**900秒 → 65秒（92.8%短縮）**
- エラー復旧時間：**15分 → 1分強へ大幅短縮**

**技術スタック**

AWS SQS（メッセージキュー）, New Relic（監視ツール）, Terraform（インフラ管理）, Kubernetes, Python

**工夫したポイント**

- 「なんとなく」ではなく、実測データに基づいて設定値を決定
- 意図的にエラーを発生させるテストで、再処理の安全性を確認
- 一気に変更するのではなく、段階的に進めてリスクを最小化
- メッセージの受信回数を記録し、リトライかどうかを自動判定する仕組みを実装
- 設定変更→監視→改善のサイクルを回せる運用フローを確立

**事業への貢献**

- エラー復旧時間の短縮により、ユーザー体験を向上
- 過剰な設定によるリソース浪費を削減
- 他の機械学習サービスにも適用できる最適化手法を確立
- システム全体の信頼性と効率性を向上

**キーワード**

### 4.7.4 **S3アップロードの並列化によるコスト削減**

**役割**：設計・実装・検証 主担当

**目的**

画像処理の成果物を保存する際、複数ファイルを並列（同時）にアップロードすることで、処理時間を短縮しコストを削減する。

**背景・課題**

- グリーティングカード用の画像切り抜き処理で、4つのファイルを1つずつ順番にアップロードしていた
- ネットワーク通信の待ち時間が積み重なり、1ジョブあたり平均8-10秒かかっていた
- 大規模なバッチ処理（300台のサーバー × 12時間稼働）では、わずかな改善でも大きなコスト削減に
- 既存のプログラムは、1ファイルずつしかアップロードできない仕組みだった

**取り組み**

1. **並列アップロード機能の実装**
    - ThreadPoolExecutor（Pythonの並列処理ライブラリ）を使用
    - 4つのファイルを同時にアップロードできる仕組みを構築
    - 並列数は4に設定（4ファイルを同時処理）
2. **AWS SDKのスレッド安全性への対応**
    - boto3（AWS SDK）は複数スレッドで共有できないため、各スレッドで個別にクライアントを作成
    - エラーが1つでも発生したら、全体を失敗とみなすシンプルな設計
3. **汎用的なAPIの設計**
    - 画像とJSON両方に対応
    - 他のサービス（ステッカー自動提案など）でも使える共通モジュールとして実装
4. **処理時間の詳細計測**
    - New Relic（監視ツール）で各ステップの処理時間を記録
    - ダウンロード、推論、生成、アップロードの各時間を可視化
    - どこがボトルネックかを特定
5. **段階的なリリース戦略**
    - Phase 1: 共通モジュールを先にリリース（他サービスでも使えるように）
    - Phase 2: グリーティングカードで実際に適用（余裕があれば実施）
    - インターン終了後も、社内メンバーが適用できる設計

**成果（定量）**

- アップロード時間：高解像度画像（6.5MB）で**0.789秒 → 0.547秒（30.7%短縮）**
- アップロード時間：通常サイズ（331KB）で**0.305秒 → 0.267秒（12.5%短縮）**
- コスト削減：グリーティングカードで**約1万円/回**、将来的にステッカーで**約10万円/回**の削減見込み

**技術スタック**

Python（ThreadPoolExecutor, boto3）, AWS S3, New Relic（監視ツール）, Kubernetes, Docker

**工夫したポイント**

- AWS SDKの特性（スレッド安全性）を理解し、適切に対応
- 他のサービスでも使える汎用的な設計
- 1枚の画像 → 複数の画像 → 本番環境の順で、段階的にテスト
- 処理時間を細かく計測し、次のボトルネック（機械学習の推論部分）を特定
- ファイルサイズが大きいほど並列化の効果が高いことを実証

**課題と解決**

1. **S3キー構造の不一致**
    - 問題：Railsアプリ側とPython側でS3キー（保存場所）の構造が違っていた
    - 解決：Rails側の実装を確認し、Python側を合わせて修正
2. **エラー時のメッセージ削除漏れ**
    - 問題：エラー発生時、SQSメッセージが削除されず、処理が完了していなかった
    - 解決：エラーハンドリング処理に必要なパラメータを追加
3. **機械学習推論の並列化は不可**
    - 問題：全体の処理時間における改善割合が小さい
    - 検討：機械学習の推論部分も並列化できないか検討
    - 結論：1回の処理につき1枚の画像なので、推論自体は並列化できない。I/O処理（アップロード）の最適化が最適解

**事業への貢献**
